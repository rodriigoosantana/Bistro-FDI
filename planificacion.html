Planificación del Proyecto Bistro FDI

Estrategia de Desarrollo

El proyecto Bistro FDI se desarrollará siguiendo una metodología ágil e incremental, dividiendo el trabajo en entregas parciales que corresponden con las prácticas del curso. Cada entrega añadirá funcionalidades y mejoras sobre la anterior, permitiendo un desarrollo iterativo con retroalimentación constante del tutor del proyecto.

La estrategia se basa en tres pilares fundamentales: comunicación continua entre los miembros del equipo, revisiones de código para mantener la calidad, y documentación progresiva que facilite el mantenimiento futuro.

Metodología de Trabajo

Organización del Equipo

El equipo se organizará de forma horizontal, con responsabilidades compartidas y rotación de roles según las necesidades de cada fase. Se establecerán reuniones semanales para coordinar el trabajo y resolver dudas.

  • Reuniones semanales: Lunes a las 18:00h (presencial o virtual)
  • Comunicación diaria: Grupo de WhatsApp/Telegram para consultas rápidas
  • Repositorio compartido: GitHub para control de versiones
  • Documentación colaborativa: Google Docs para documentos compartidos

Distribución de Tareas

Las tareas se asignarán considerando las fortalezas de cada miembro y buscando que todos participen en diferentes aspectos del desarrollo (frontend, backend, base de datos, documentación).

Criterios de asignación:

  • Experiencia previa y conocimientos de cada miembro
  • Equilibrio en la carga de trabajo
  • Oportunidades de aprendizaje en nuevas tecnologías
  • Preferencias personales cuando sea posible

Control de Versiones

Se utilizará Git como sistema de control de versiones y GitHub como plataforma de alojamiento del repositorio. Se seguirán las siguientes prácticas:

  • Rama principal (main): Código estable y funcional
  • Rama de desarrollo (develop): Integración de nuevas funcionalidades
  • Ramas de características (feature/): Desarrollo de funcionalidades específicas
  • Commits descriptivos: Mensajes claros que expliquen los cambios realizados
  • Pull requests: Revisión de código antes de fusionar cambios

Fases del Proyecto

Fase 0: Formación del Grupo y Planificación Inicial

Período: Hasta mediados de enero 2026

Objetivos:

  • Constitución formal del grupo de trabajo
  • Lectura y análisis del enunciado del proyecto
  • Identificación de requisitos funcionales y técnicos
  • Discusión del alcance con el tutor del proyecto
  • Establecimiento de canales de comunicación
  • Creación del repositorio en GitHub

Entregables:

  • Documento de constitución del grupo
  • Repositorio GitHub inicializado
  • Acta de reunión con el tutor

Fase 1: Documentación y Diseño (Práctica 1)

Período: Enero - 13 de febrero 2026

Objetivos:

  • Diseñar la estructura de la aplicación
  • Crear bocetos de las interfaces principales
  • Documentar funcionalidades detalladamente
  • Planificar el resto del proyecto
  • Crear sitio web descriptivo del proyecto

Tareas específicas:

  • Diseño de bocetos en papel para todas las pantallas principales
  • Definición de flujos de navegación entre pantallas
  • Descripción detallada de cada funcionalidad
  • Creación de páginas HTML: index, detalles, bocetos, miembros, planificación, contacto
  • Diseño del logo e imágenes descriptivas
  • Validación HTML5 de todas las páginas

Entregables:

  • Sitio web descriptivo completo (p1_g<numGrupo>.zip)
  • Bocetos de todas las interfaces
  • Documento de planificación detallada

Distribución tentativa de tareas:

  • Miembro 1: index.html y detalles.html
  • Miembro 2: bocetos.html y diseño de bocetos
  • Miembro 3: miembros.html y recopilación de información del equipo
  • Miembro 4: planificacion.html y contacto.html
  • Todos: Revisión cruzada y validación HTML

Fase 2: Prototipo Funcional (Práctica 2)

Período: Febrero - 6 de marzo 2026

Objetivos:

  • Diseñar la arquitectura del sistema
  • Diseñar y crear la base de datos
  • Implementar HTML básico (sin CSS) de las vistas principales
  • Implementar funcionalidades 0, 1 y 2 (grupos de 4)
  • Implementar funcionalidades 0, 1, 2 y 3 (grupos de 5)
  • Configurar control de versiones en GitHub

Tareas específicas:

  • Diseño del esquema de base de datos (ERD)
  • Creación de scripts SQL (schema.sql y seed.sql)
  • Implementación del sistema de usuarios y autenticación
  • Desarrollo del CRUD de productos y categorías
  • Implementación del flujo completo de pedidos
  • Creación de vistas PHP para cada funcionalidad
  • Separación de lógica de negocio y presentación
  • Pruebas funcionales de cada módulo

Entregables:

  • Código fuente en GitHub
  • Base de datos funcional con datos de prueba
  • Aplicación parcialmente funcional (≈50% de funcionalidades)
  • Documento de arquitectura del sistema

Distribución tentativa de tareas:

  • Miembro 1: Base de datos y funcionalidad 0 (usuarios)
  • Miembro 2: Funcionalidad 1 (gestión de productos)
  • Miembro 3 y 4: Funcionalidad 2 (gestión de pedidos - cliente)
  • Miembro 5: Funcionalidad 3 (preparación de pedidos) [si grupo de 5]
  • Todos: Integración y pruebas

Fase 3: Mejora Visual y Funcional (Práctica 3)

Período: Marzo - 8 de abril 2026

Objetivos:

  • Implementar diseño CSS completo
  • Mejorar la experiencia de usuario
  • Añadir funcionalidad 3 (grupos de 4) o funcionalidad 4 (grupos de 5)
  • Desplegar la aplicación en VPS (obligatorio)
  • Alcanzar aproximadamente el 75% de funcionalidades completas

Tareas específicas:

  • Diseño de hojas de estilo CSS responsive
  • Adaptación a diferentes dispositivos (móvil, tablet, escritorio)
  • Implementación de funcionalidades adicionales
  • Mejora de la interfaz de usuario
  • Optimización de consultas a la base de datos
  • Configuración y despliegue en servidor VPS
  • Pruebas en diferentes navegadores
  • Corrección de errores detectados

Entregables:

  • Aplicación con diseño CSS completo
  • Aplicación desplegada y accesible en VPS
  • ≈75% de funcionalidades implementadas
  • Manual de instalación en VPS

Distribución tentativa de tareas:

  • Miembro 1 y 2: Diseño y desarrollo CSS
  • Miembro 3: Funcionalidad 3 o 4 según tamaño del grupo
  • Miembro 4: Despliegue en VPS y configuración
  • Miembro 5: Pruebas y corrección de errores [si grupo de 5]
  • Todos: Testing en diferentes dispositivos

Fase 4: Finalización y Documentación (Entrega Final)

Período: Abril - 3 de mayo 2026

Objetivos:

  • Completar todas las funcionalidades pendientes
  • Implementar funcionalidad 4 (grupos de 4) o funcionalidad 5 (grupos de 5)
  • Añadir mejoras opcionales (AJAX, JSON, servicios web)
  • Realizar pruebas exhaustivas
  • Elaborar la memoria completa del proyecto
  • Preparar datos de prueba detallados
  • Optimizar el rendimiento

Tareas específicas:

  • Implementación de funcionalidades restantes
  • Implementación de mejoras opcionales (AJAX, servicios web, etc.)
  • Validación y sanitización de todos los inputs
  • Implementación de seguridad (hash de contraseñas, prevención SQL injection)
  • Optimización de rendimiento y consultas
  • Creación de scripts de base de datos completos
  • Elaboración de la memoria descriptiva
  • Preparación de manual de usuario
  • Testing completo de la aplicación
  • Corrección de bugs finales

Entregables:

  • Aplicación web 100% funcional
  • Memoria descriptiva completa
  • Scripts de base de datos (schema + seed)
  • Manual de instalación y uso
  • Aplicación desplegada en VPS
  • Archivo comprimido: proyecto_g<numGrupo>.zip

Distribución tentativa de tareas:

  • Miembro 1: Funcionalidades finales y mejoras opcionales
  • Miembro 2: Seguridad y optimización
  • Miembro 3: Memoria descriptiva y documentación técnica
  • Miembro 4: Testing y corrección de errores
  • Miembro 5: Manual de usuario y datos de prueba [si grupo de 5]
  • Todos: Revisión final y preparación de la entrega

Fase 5: Preparación del Examen Individual

Período: Mayo (previo al 13 de mayo 2026)

Objetivos:

  • Repasar todo el código desarrollado
  • Asegurar que todos los miembros comprenden todas las partes
  • Practicar la implementación de nuevas funcionalidades
  • Prepararse para el examen práctico individual

Tareas:

  • Sesiones de repaso conjuntas
  • Explicaciones cruzadas entre miembros
  • Ejercicios prácticos de extensión del proyecto
  • Revisión de conceptos clave del curso

Gestión de Riesgos

Riesgos Identificados y Mitigación

1. Disponibilidad de los miembros del equipo

Riesgo: Diferencias de horarios y disponibilidad entre miembros.

Mitigación: Establecer horarios de reunión desde el inicio, comunicación asíncrona efectiva, y documentación clara de todas las decisiones.

2. Dificultades técnicas

Riesgo: Bloqueos por falta de conocimiento en alguna tecnología.

Mitigación: Consultar documentación oficial, tutoriales, foros especializados y, si es necesario, contactar con el tutor del proyecto.

3. Conflictos en el control de versiones

Riesgo: Conflictos de merge en GitHub.

Mitigación: Commits frecuentes, pull frecuente de cambios, comunicación antes de modificar archivos compartidos, y uso adecuado de ramas.

4. Subestimación del tiempo necesario

Riesgo: No llegar a tiempo a las entregas.

Mitigación: Planificación realista con margen de seguridad, seguimiento semanal del progreso, y priorización de funcionalidades core.

5. Desigualdad en la carga de trabajo

Riesgo: Distribución desigual del esfuerzo entre miembros.

Mitigación: Revisión periódica de las contribuciones, rotación de tareas, y uso de GitHub para monitorizar aportaciones.

Criterios de Calidad

Para asegurar la calidad del proyecto, se seguirán los siguientes criterios:

  • Código limpio: Indentación correcta, nombres descriptivos, comentarios apropiados
  • Validación: HTML5 válido, CSS válido, JavaScript sin errores
  • Seguridad: Sanitización de inputs, hash de contraseñas, prevención de SQL injection
  • Arquitectura: Separación clara entre capas (modelo-vista-controlador)
  • Usabilidad: Interfaces intuitivas, mensajes de error claros, feedback al usuario
  • Responsividad: Adaptación a diferentes tamaños de pantalla
  • Documentación: Código documentado, README actualizado, memoria completa
  • Testing: Pruebas funcionales de cada módulo antes de integrar

Plataformas y Servicios

  • VPS: Servidor proporcionado por la universidad

  • Google Drive/Docs: Documentación colaborativa
  • WhatsApp/Telegram: Comunicación rápida del equipo

  • HTML5 y CSS3

  • PHP y programación orientada a objetos

  • Git y GitHub
  • Configuración básica de servidores Linux

Conclusión

Esta planificación establece un marco de trabajo claro y estructurado para el desarrollo exitoso del proyecto Bistro FDI. El enfoque incremental permite validar cada fase antes de avanzar a la siguiente, reduciendo riesgos y facilitando la detección temprana de problemas.

El compromiso de todos los miembros del equipo, la comunicación efectiva y el seguimiento riguroso de esta planificación serán claves para alcanzar los objetivos propuestos y obtener una calificación satisfactoria en el proyecto.
